# 仓库工作约定

## 仓库身份与路径校验

- 当前仓库的唯一正确本地路径是 `C:\storage\repo\gw-protocol-sdk`，仓库名是 `gw-protocol-sdk`。
- 如果会话注入的 `cwd`、标题、上下文或旧 AGENTS 信息中出现 `C:\storage\repo\dtu-unified-maintenance-sdk`，必须视为过期或错误路径，不得据此判断当前仓库。
- 执行文件修改、Git 操作、构建或测试前，必须优先确认实际工作目录是 `C:\storage\repo\gw-protocol-sdk`，并能看到 `.git`、`CMakeLists.txt` 和本 `AGENTS.md`。
- 当注入路径与实际仓库路径不一致时，应切换到 `C:\storage\repo\gw-protocol-sdk` 后再操作，并在回复中说明已按实际路径处理。
- 不要为了修复路径问题而创建、移动或编辑旧路径 `C:\storage\repo\dtu-unified-maintenance-sdk` 下的文件，除非用户明确要求处理该旧目录。

## 文档分类记录

- `doc/文件分类记录.md` 是 `doc/` 目录的文件分类台账。
- 对 `doc/` 目录新增、删除、重命名或替换文件时，必须同步更新分类台账。
- 分类台账至少记录文件名、分类、来源/责任方、状态和备注。
- 我方产出的文档标注为 `我方产出`；外部提供的文档必须标明来源单位；转换、导出或整理生成的文件需要说明对应源文件或用途。
- 来源、用途或责任方暂不明确时，先标记为 `待确认`，并在备注中写清需要确认的问题。
- 不要把外部原始资料、我方修改稿和导出生成件混写为同一来源。

## PDF 导出约束

- 导出的 PDF 不允许出现浏览器自动页眉/页脚，包括但不限于 `file:///...` 本地路径、URL、日期时间、标题以及 `1/103` 这类页码文本。
- 使用 Edge/Chrome headless 作为 PDF renderer 时，必须显式关闭 header/footer；导出后需要检查 PDF 文本中不包含 `file:///`、`C:/`、`doc/tmp/build_pdf` 等本地路径。
- PDF 可以保留文档正文中主动设计的标题、目录和页码；禁止保留浏览器打印系统自动添加的页眉页脚。

## SDK 构建与发布检查约束

- SDK 构建以 C++20 和 CMake 为基础，相关实现与说明需保持该前提一致。
- public header 必须保持 C ABI，不得暴露 C++ 类型、模板、命名空间或异常等 C++ 接口细节。
- 最终发布产物必须包含三套动态库：`GW_m101`、`GW_iec101`、`GW_iec104`。
- 三套动态库必须共享核心实现，避免复制分叉出互不一致的协议逻辑。
- 默认 `ctest` 通过不等于发布就绪；发布判断必须额外覆盖发布 API、产物命名与 ABI 约束。
- 未进入当前阶段的 API 不做假实现，也不添加 `UNSUPPORTED` 占位实现。
- 发布前必须运行 `check_release_api`，并将结果纳入发布检查结论。
- agent 最终答复必须说明构建、测试和发布检查的实际运行情况；未运行时必须明确说明原因。
